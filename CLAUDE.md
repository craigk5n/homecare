## What this is

HomeCare is a small PHP + MySQL web app for tracking a patient's medication schedule, intake, inventory, and caregiver notes. It is a single-purpose internal tool served directly by Apache from `/var/www/html/homecare` (no build step, no package manager, no test suite). Several infrastructure files (`includes/init.php`, `includes/dbi4php.php`, `includes/functions.php`, `includes/translate.php`, `includes/formvars.php`, `includes/menu.php`, `pub/`) are borrowed from [WebCalendar](https://github.com/craigk5n/webcalendar) and follow its conventions.

## Architecture

### Request model
Every user-facing URL is a top-level `*.php` file in the repo root. There is no router and no framework. The standard page shape is:

```php
require_once 'includes/init.php';  // bootstrap: config, DB, translation, helpers
print_header();                    // emits <html>/<head>/<body> + Bootstrap chrome + menu
// ... render page body, usually with dbi_* calls and inline HTML ...
echo print_trailer();              // closes the page and closes the DB connection
```

`includes/init.php` enforces that files under `includes/` cannot be requested directly. `print_header()` and `print_trailer()` live in `init.php` and pull heavily from globals defined in `includes/config.php`.

### Module layout
- **`includes/init.php`** ‚Äî single bootstrap included by every page. Pulls in config, the DB layer, form helpers, translation, validation, and domain code. Owns `print_header` / `print_trailer`.
- **`includes/config.php`** ‚Äî loads `includes/settings.php` (DB credentials, written by the WebCalendar-style installer), defines globals, calls `do_config()` on boot.
- **`includes/settings.php`** ‚Äî site-local secrets (DB host/user/password/type). Generated by installer; not meant for version control.
- **`includes/dbi4php.php`** ‚Äî generic DB abstraction copied from WebCalendar. Use its API (`dbi_query`, `dbi_fetch_row`, `dbi_get_cached_rows($sql, [$params])`, `dbi_error`, `dbi_close`) rather than raw `mysqli_*`. Parameterised queries go through the `$params` array argument ‚Äî **always use it for user input**, never interpolate.
- **`includes/homecare.php`** ‚Äî all HomeCare-specific domain logic: `getPatient`, `getPatients`, `calculateSecondsUntilDue`, `calculateNextDueDate`, `frequencyToSeconds`, `dosesRemaining`, `getDueDateTimeInSeconds`, `getInventoryDashboardData`. New domain functions belong here, not in `functions.php` (which is the generic WebCalendar utility dumping ground).
- **`includes/formvars.php`** ‚Äî `getGetValue`, `getPostValue`, `getValue` helpers. Use these to read request input instead of touching `$_GET`/`$_POST` directly, so the WebCalendar validation layer applies.
- **`includes/translate.php`** ‚Äî wraps user-visible strings via `translate()`. Language files live in `translations/`.
- **Root `*.php` files** ‚Äî each is either a page (`list_medications.php`, `list_schedule.php`, `edit_medication.php`, `inventory_dashboard.php`, `inventory_refill.php`, `inventory_adjust.php`, `inventory_history.php`, `merge_medicines.php`, `report_*.php`, `schedule_daily.php`), a form POST handler (`*_handler.php`), or a CLI/cron script (`send_reminders.php`). `edit_inventory.php` redirects to `inventory_dashboard.php`.

### Frequency convention
Medication frequency is stored as a short string like `8h`, `12h`, `1d`, `30m`. Always parse it via `frequencyToSeconds()` / `getIntervalSpecFromFrequency()` / `calculateSecondsUntilDue()` in `includes/homecare.php` ‚Äî these are the canonical parsers and they throw on unknown units.

### Database
MySQL (via the `mysqli` driver set in `settings.php`). All app tables are prefixed `hc_`. Canonical schema is in `tables-mysql.sql`:

- `hc_patients` ‚Äî patient roster (`is_active` flag for soft-disable)
- `hc_medicines` ‚Äî **product catalog** (name + dosage form/strength). Does **not** contain `frequency` or `unit_per_dose` ‚Äî those belong on schedules.
- `hc_medicine_schedules` ‚Äî a patient's prescription: links a product to a patient with dosing instructions. `frequency` and `unit_per_dose` are non-nullable and authoritative here (no fallback to medicine-level defaults).
- `hc_medicine_intake` ‚Äî each recorded dose taken (FK ‚Üí schedule)
- `hc_medicine_inventory` ‚Äî stock snapshots per medicine/product (`current_stock` + timestamp). Tracks at the product level.
- `hc_caregiver_notes` ‚Äî free-text notes per patient
- `hc_user`, `hc_config` ‚Äî auth + settings, WebCalendar-style

Remaining-doses math in `dosesRemaining()` works by taking the most recent inventory row and subtracting `SUM(s.unit_per_dose)` of intakes (from `hc_medicine_schedules`) since that inventory timestamp ‚Äî so inventory rows are checkpoints, not running counters. Keep it that way when touching inventory logic. `unit_per_dose` is always read from the schedule, never from the medicine.

### External integrations
- **`schedule_ics.php`** emits an iCalendar feed of today/tomorrow doses for a given `?patient_id=‚Ä¶`. Caregivers subscribe to this URL from their calendar app.
- **`send_reminders.php`** is a CLI/cron script that pushes due-soon alerts to an [ntfy](https://ntfy.sh/) topic. The ntfy URL and channel are currently hard-coded at the top of the file (marked `TODO: move to hc_config`). Supports `--dry-run` and an optional numeric arg for "minutes before due" (default 5).

## Common commands

There is no `composer`, `npm`, `make`, or test runner in this repo. Operations are manual:

```bash
# Apply / reset the schema (destructive):
mysql intranet < tables-mysql.sql

# Run pending migrations (non-destructive, backup first):
./dump.sh
mysql intranet < migrations/001_separate_product_prescription.sql

# Back up just the HomeCare tables into homecare-dump.sql:
./dump.sh

# Trigger medication reminders (cron entry point):
php send_reminders.php              # send real notifications, 5-min window
php send_reminders.php --dry-run    # print what would be sent
php send_reminders.php 15           # use a 15-minute due window
```

The DB name (`intranet`) and credentials come from `includes/settings.php`, which is environment-specific and not in source control.

## Conventions to follow when editing

- **Pages follow the `init.php` ‚Üí `print_header()` ‚Üí body ‚Üí `print_trailer()` shape.** Don't introduce a router or an autoloader for one-off pages.
- **Always use `dbi_*` with parameter arrays** ‚Äî e.g. `dbi_get_cached_rows($sql, [$patient_id, $schedule_id])`. Never string-concatenate user input into SQL.
- **Always HTML-escape output** with `htmlspecialchars(...)` when echoing values from the DB or request (see `list_medications.php` for the pattern).
- **Read request input via `getGetValue` / `getPostValue`** from `formvars.php`, not raw superglobals.
- **New HomeCare-specific helpers go in `includes/homecare.php`.** Leave `functions.php` / `dbi4php.php` / `translate.php` alone unless you are intentionally pulling a fix back from upstream WebCalendar.
- **Frequencies** are short strings (`Nd` / `Nh` / `Nm`) ‚Äî parse via the helpers in `homecare.php`, don't re-invent.
- **Globals are real here.** `init.php` and `config.php` expose dozens of `$GLOBALS['‚Ä¶']` values that pages rely on (e.g. `$login`, `$c`, `$SQLLOG`, `$PROGRAM_VERSION`). Don't rename them casually.
